GitHubUpdater SwiftUI Update Window — Implementation Plan

Generated: 2026-06-27
Source: Docs/agent-plans/2026-06-27-updater/updater.html

Overview

Today GitHubUpdater reports the “update available” outcome with a plain NSAlert (View and Download / Later) that opens the release page in the default browser. This plan replaces only that outcome with a dedicated, SwiftUI-based update window that renders the release’s Markdown notes and offers richer actions (Download, View on GitHub, Later). The up-to-date and error outcomes are out of scope and keep their existing modal alerts.

The approach works outward from the data toward the UI: first carry the release notes and a direct download link through the platform-agnostic check (the current parser captures only the tag and the release-page URL), then add a Markdown renderer, then build the SwiftUI content, and finally host it in an AppKit window and route the outcome to it. Each milestone keeps the package building and its tests green so it can be reviewed and committed on its own.

Questions & answers

Which source document should the plan be built from?
Docs/agent-plans/2026-06-27-updater/updater.html (the recommended candidate under Docs/agent-plans).
What output format should the plan use?
HTML, matching the source document’s format.
How should the release notes’ Markdown be rendered? (The package currently has zero dependencies.)
After discussing the trade-offs — a single SwiftUI Text renders inline styling but ignores block structure; a hosted NSTextView can render blocks once intents are converted to concrete attributes; and Apple’s parsers give semantic intent rather than ready-to-display styling — the chosen approach is to add Apple’s swift-markdown and render its AST with custom SwiftUI views. This gives GitHub-Flavored-Markdown fidelity (tables, task lists, strikethrough) and clean handling of nested structure, at the cost of one Apple-maintained dependency.
The “Download” button should open the release’s direct download link. Which asset?
The first release asset’s browser_download_url.
If a release has no downloadable asset, what should the Download button do?
Hide the Download button; the window shows only View on GitHub and Later.

Notes

  • Key files. SwiftUtilities/GitHubUpdater.swift (platform-agnostic check + release parsing), SwiftUtilities/UpdateCheckResult.swift (the result enum), SwiftUtilities/GitHubUpdater+AppKit.swift (AppKit presentation, MessageOptions, the internal Alert routing), SwiftUtilities/Localization.swift + SwiftUtilities/en.lproj/Localizable.strings (localized text), and the mirrored test files under SwiftUtilitiesTests/.
  • Conventions to follow. Keep the existing license header on every new file; document every type and member (public and private) with SwiftDoc; prefer let and higher-order functions over raw loops; align consecutive = signs; separate calls from assignments with a blank line; use early returns. Run the repo’s Xcode-Format pass on changed files. All user-facing strings go through Localization.string(_:) and Localizable.strings.
  • Platform guards. The AppKit/SwiftUI code stays behind #if canImport( AppKit ) (and canImport( SwiftUI ) where needed), consistent with GitHubUpdater+AppKit.swift. swift-markdown is a cross-platform Swift package added to the target unconditionally; it is only imported and used from guarded files.
  • API-change cascade (important for ordering). Adding associated values to UpdateCheckResult.updateAvailable breaks every pattern-match on it — the internal Alert.updateAvailable case, alert(for:messages:), present(_:messages:), and the AppKit tests. Milestone 1 therefore threads the new fields through all of those to keep the build and tests green, even though presentation is still the old NSAlert at that point.
  • Window lifetime. GitHubUpdater is a Sendable final class and present(_:messages:) is @MainActor. To avoid adding mutable state to the updater, the window is owned by a dedicated @MainActor controller that self-retains until its window closes (via NSWindowDelegate). The window is non-modal, unlike the alert it replaces.
  • Testing reality. The pure value logic stays unit-tested (parsing, result mapping, alert routing). SwiftUI views and AppKit window hosting are verified manually; where a Markdown AST→model mapping is factored out as pure, it should be unit-tested too.
  • Open question for review. Whether to delete the now-unused GitHubUpdater.alert.updateAvailable.* strings (title / message / buttons) once the window replaces the alert, or keep them. Decided during Milestone 4.

Milestones

Phase 1 · Data

  1. 1 Carry release notes and download link through the update check

    Goal

    Extend the platform-agnostic check so it captures the release’s Markdown body and a direct download URL (the first asset), in addition to the release-page URL it captures today. The new data flows into UpdateCheckResult.updateAvailable so the window has what it needs. Presentation is unchanged in this milestone; the build and all tests stay green.

    Tasks
    • Extend GitHubUpdater.parseReleases(from:) to also read body (the Markdown notes, defaulting to an empty string when absent) and the first element of the release’s assets array’s browser_download_url (an optional URL string). Widen its return tuple accordingly (e.g. (version, notes, releaseURL, downloadURL)).
    • Update GitHubUpdater.updateCheckResult(current:program:releases:) to thread notes and downloadURL from the newest release into the result.
    • Add notes: String and downloadURL: URL? associated values to UpdateCheckResult.updateAvailable; update its doc comment to describe them.
    • Update the cascade in GitHubUpdater+AppKit.swift: extend the internal Alert.updateAvailable case and alert(for:messages:) to carry the new fields, and adjust present(_:messages:) / showUpdateAvailableAlert(...) signatures so it still compiles (the alert may ignore the new fields for now).
    • Update tests: in SwiftUtilitiesTests/GitHubUpdater.swift assert that parseReleases captures notes and the first asset’s download URL, that a missing assets/browser_download_url yields nil, and that a missing body yields an empty string; assert updateCheckResult propagates both. In SwiftUtilitiesTests/GitHubUpdater+AppKit.swift update the .updateAvailable constructions/expectations to the new signature.
    • Run swift build and swift test; confirm green.

    Implementation notes
    • Parser. GitHubUpdater.parseReleases(from:) now returns the widened tuple (version: String, notes: String, url: String, downloadURL: String?). notes reads the release’s body, defaulting to "" when absent. downloadURL reads assets[0].browser_download_url, staying nil when the assets key is missing, the array is empty, or the first asset has no browser_download_url. The existing release-page URL keeps the name url (rather than renaming to releaseURL) to avoid churning the unrelated parseReleases sorting tests.
    • Result mapping. updateCheckResult(current:program:releases:) takes the widened tuple and threads notes through verbatim; downloadURL is converted from String? to URL? with latest.downloadURL.flatMap { URL( string: $0 ) }, so an unparseable asset URL simply becomes nil rather than failing the whole check (only the release-page url failing still produces .failed).
    • Result enum. UpdateCheckResult.updateAvailable gained notes: String and downloadURL: URL? associated values, with the doc comment describing both.
    • AppKit cascade. The internal Alert.updateAvailable case, alert(for:messages:), the present(_:messages:) routing, and showUpdateAvailableAlert(...) now all carry notes and downloadURL. Presentation is unchanged for this milestone — the alert ignores the two new fields (documented as “carried for the forthcoming update window”).
    • Tests (TDD). Wrote the new assertions first and watched them fail to compile (missing tuple members), then implemented. Added parser tests for notes capture, first-asset download URL, empty-body default, and the three no-download-URL shapes; added updateCheckResult propagation tests (with and without a download URL); added a performUpdateCheck end-to-end test carrying notes + download URL. Updated all existing updateCheckResult / .updateAvailable / Alert.updateAvailable constructions to the new signatures. Introduced a private ParsedRelease typealias in the test struct so the tuple literals (whose downloadURL: nil needs a contextual type) stay readable.
    • Verification. swift build and swift test both green — 66 tests pass. (Editor SourceKit may still show stale errors against the old tuple type; the compiler is clean.)

Phase 2 · Rendering & UI

  1. 2 Add swift-markdown and a Markdown-rendering SwiftUI view

    Goal

    Bring in Apple’s swift-markdown and build a reusable SwiftUI view that renders a Markdown string as native SwiftUI views — headings, paragraphs, emphasis/strong, inline code, code blocks, links, ordered and unordered (including nested) lists, block quotes, and thematic breaks — degrading gracefully for unsupported nodes.

    Tasks
    • Add .package(url: "https://github.com/apple/swift-markdown.git", ...) to Package.swift with a pinned version, and add the Markdown product to the SwiftUtilities target’s dependencies. Commit the resulting Package.resolved.
    • Create a new guarded source file (e.g. SwiftUtilities/GitHubUpdater+MarkdownView.swift) under #if canImport( SwiftUI ), importing Markdown and SwiftUI.
    • Implement a MarkupVisitor (or walker) that maps AST nodes to a small view model / SwiftUI views: inline runs (text, emphasis, strong, inline code, links) composed into Text/AttributedString; block nodes laid out in a VStack.
    • Make links tappable, opening the target in the default browser via NSWorkspace.shared.open.
    • Factor the AST→view-model mapping so the pure part is unit-testable; add tests for representative inputs (heading, nested list, link, code block). Note that the rendered views themselves are verified manually.
    • Document every new type and member (public and private) with SwiftDoc. Run swift build.

    Goal

Appendix

GitHub release JSON fields used (from the releases endpoint):

FieldUseStatus
tag_nameVersion comparison (normalized, v-prefix stripped).Captured today.
html_urlRelease page — the “View on GitHub” action.Captured today.
draft / prereleaseEntries with either set to true are discarded.Captured today.
bodyMarkdown release notes rendered in the window.Added in Milestone 1.
assets[0].browser_download_urlDirect download link — the “Download” action (button hidden if absent).Added in Milestone 1.

Data flow after the change:

GitHub releases JSON
  → parseReleases(from:)            // + body, + assets[0].browser_download_url
  → updateCheckResult(...)          // threads notes + downloadURL
  → UpdateCheckResult.updateAvailable(application, version, update, url, notes, downloadURL)
  → [AppKit] alert(for:messages:)   // Alert.updateAvailable carries the same fields
  → present(_:messages:)            // .updateAvailable opens the window (was NSAlert)
  → UpdateWindowController          // @MainActor, self-retaining NSWindow
       hosts UpdateAvailableView (SwiftUI)
         → MarkdownView(notes)       // swift-markdown AST → custom SwiftUI views
         → [Download?] [View on GitHub] [Later]

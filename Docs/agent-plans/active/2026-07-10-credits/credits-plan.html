Licenses & Credits Window

Milestone-based implementation plan

Generated: 2026-07-10 Source: Docs/agent-plans/active/2026-07-10-credits/credits.md Target: SwiftUtilities (macOS 15, SwiftUI + AppKit)

Overview

Add a reusable Credits window to the SwiftUtilities library that a consumer application can configure and present to display the licenses and credits of the third-party projects it uses. The window is a two-pane split view: a sidebar listing every credited project alphabetically (name, author, and a license pill), and a detail pane showing the selected project's name, license pill, author, description, website link, and its full license text in a scrollable area.

The approach mirrors the library's existing conventions established by the About window: purely presentational SwiftUI views that receive their data through initializers (no bundle coupling), a reusable public component for the license pill, a plain public value type describing one credited project, and a thin HostingWindowController subclass that exposes a static show(credits:) entry point and inherits the single-window lifetime management. Work is split bottom-up — leaf component first, then the model, the views, and finally the window controller and public entry point — so each milestone builds on a reviewed foundation and can be committed on its own.

Questions & answers

How should the public API and types be named for this feature?
Use Credits naming: CreditsWindowController, CreditsView, and a Credit model type — concise and consistent with the existing About* naming.
The license name is shown as a pill in two places (sidebar + detail). How should that pill be implemented?
As a reusable public component (a small Pill view) usable across apps and reused in both spots here, matching the library's public-first convention.
Besides the explicit API where the consumer builds the credit items in code, should the plan include a bundle-loading convenience?
Explicit API only. Ship just the show(credits:) path, mirroring the core AboutWindowController.show(applicationName:…). Any resource-loading helper is deferred.

Notes

Conventions to follow (from the existing codebase and CLAUDE.md):

  • Every source file starts with the project's MIT license header block (copy it verbatim from an existing file such as SwiftUtilities/About/AboutView.swift).
  • One type per file; group the feature's files under a new SwiftUtilities/Credits/ folder, and place the reusable pill under a new SwiftUtilities/Components/ folder.
  • Pure SwiftUI files are gated with #if canImport( SwiftUI ); files touching AppkKit (the window controller) use #if canImport( AppKit ), as AboutWindowController does.
  • Presentational views take their data through public inits and expose no bundle dependency (see AboutView, UpdateAvailableView). Every view has a #Preview.
  • Prefer public for consumer-facing types, let over var, early returns, no force-unwraps, and forEach/map over raw loops. Document public and private members with doc comments. Align consecutive = signs.
  • User-facing strings go through Localization.string(_:) with keys added to SwiftUtilities/Utilities/en.lproj/Localizable.strings (with a descriptive comment), following the existing About.window.title / GitHubUpdater.* entries.
  • Keep the Xcode project and the SPM manifest in sync. Each new file must be added to the SwiftUtilities target in SwiftUtilities.xcodeproj; the new folders are already covered by the SPM path: "SwiftUtilities" target, so Package.swift needs no change (no new resources are introduced). Prefer building/testing via xcodebuild over swift build.

Key existing files to model after:

  • SwiftUtilities/About/AboutView.swift — presentational view shape & preview.
  • SwiftUtilities/About/AboutWindowController.swift — show(…) entry point, window chrome.
  • SwiftUtilities/Windows/HostingWindowController.swift — single-window base, Sizing.
  • SwiftUtilities/GitHub/UpdateAvailableView.swift — scrollable area with a distinct background/border (reuse for the license text).
  • SwiftUtilities/Utilities/Localization.swift + en.lproj/Localizable.strings.

Design decisions taken as defaults (flag during review if undesired):

  • The Credit model carries: project name, author, project description, an optional website URL, the license name (for the pill), and the full license text. It is Identifiable + Hashable so it can drive a NavigationSplitView selection.
  • Alphabetical ordering of the sidebar is done by the view (case-insensitive, by project name); the consumer can pass the credits in any order.
  • Unlike the fixed-size About window, the Credits window is resizable with a sensible minimum content size; it opens at a fixed initial size and is centered. The first project is selected by default, with a placeholder shown if the credits list is empty.
  • Description and license text are rendered as plain text (the source doc does not call for Markdown); the website is a SwiftUI Link.

Implementation notes

Created SwiftUtilities/Credits/Credit.swift: a public struct Credit: Identifiable, Hashable (MIT header, Foundation import). It stores six public let properties — name, author, description, website: URL?, licenseName, and licenseText — with a public memberwise-style init and a doc comment on every property, the id, and the init.

Identity/hashing design: id is a computed property returning the project name (a stable identifier that suits a NavigationSplitView selection), while Equatable/Hashable are synthesized over the six stored properties, so equality reflects content. As id is not stored, it does not participate in the synthesized equality.

TDD: wrote SwiftUtilitiesTests/Credits/Credit.swift (struct Test_Credit) first with a makeCredit helper and cases covering construction (all fields + missing website), identity (id == name), and equality/hashing (equal values compare equal and hash equally, differing values compare unequal). Ran xcodebuild test and confirmed the red state (cannot find type 'Credit'), then implemented the model to reach green.

Registered both files in the Xcode project: the model in the SwiftUtilities target under a new Credits group, and the test in the SwiftUtilitiesTests target under a matching Credits group. No Package.swift change was needed (both folders are covered by the existing target paths). Full suite: TEST SUCCEEDED, 82 tests / 9 suites.

Updater — Skip Version & Deployment-Target Verification

Generated: 2026-07-13

Source: Docs/agent-plans/active/2026-07-13-updater/updater.html

Overview

This plan implements the two backlog items for the in-app updater. The first lets the user skip a specific update so the "update available" window stops appearing for that version, while a strictly newer version still re-triggers it. The second verifies the downloaded app's deployment target against the host OS before the running application is replaced, aborting with a readable error when the host is too old to run the update.

The approach follows the existing updater architecture: pure, injectable building blocks with value-level decision seams that are unit-testable without UI, the network, or a live XPC connection. The skip decision reuses the module's existing numeric version comparison; the deployment-target check is inserted into the service-side install composition where the extracted bundle already exists, and surfaces through the failure path that is already wired into the window.

Questions & answers

When a version is skipped, should an explicit "Check for Updates…" still surface it, or should skipping hide that version everywhere until a newer one ships?
Manual check overrides skip. Background/automatic checks stay silent for the skipped version; an explicit user-initiated check still shows it (Sparkle-like behavior).
How should the skipped version be stored? (The codebase currently has no persistence layer at all.)
UserDefaults, single latest version. A small UserDefaults-backed helper stores one skipped-version string; skipping again replaces it. The "newer supersedes" rule makes a single value sufficient.
Where should the deployment-target (host-OS compatibility) check run?
Service-side, at install time. Read LSMinimumSystemVersion from the extracted bundle between extract and replace — the only place the bundle exists — and abort before replacing if the host OS is too old.
When the update can't run on the host OS, what should the window offer besides the error?
Error + link to release page. The failure state already renders a "View on GitHub" action alongside the message, so this is satisfied by the existing failure UI (see Notes).

Notes

  • No persistence exists yet. There is no UserDefaults, @AppStorage, or settings type anywhere in the repo. The skipped-version store is the first persisted setting; keep it small and self-contained.
  • Version comparison. Reuse GitHubUpdater.isVersion(_:newerThan:) (SwiftUtilities/Updater/GitHub/GitHubUpdater.swift:325), a normalized numeric NSString comparison (so 1.10 > 1.9). Do not introduce a second version type. The skip suppression rule is: suppress iff the available update is not strictly newer than the skipped version.
  • Manual vs. background. The two entry points differ only by their messages set: checkForUpdates() uses .all, checkForUpdatesInBackground() uses .updateAvailable (GitHubUpdater+AppKit.swift:177, :193). The skip gate must apply to the background path only — thread an explicit flag rather than inferring it from the messages set.
  • Failure UI already links to the release. In the .failed state the view renders failure(message:) plus the actions row, which includes "View on GitHub" (opens the release page) and "Later" (UpdateAvailableView.swift:190, :354). A deployment-target error that surfaces through this path needs no new UI to satisfy the "error + link" answer.
  • Errors cross the XPC boundary as text. Any LocalizedError thrown inside the install composition has its errorDescription carried through UpdateInstallResult.failure(from:) → the client → the in-window failure state. So the deployment-target error must be a LocalizedError with a clear message.
  • Single install-composition wiring site. UpdateInstallation is constructed only at UpdaterService.swift:145. That is the one place to inject the new verifier.
  • Host OS & deployment target. Prefer building an OperatingSystemVersion from the parsed LSMinimumSystemVersion and calling ProcessInfo.processInfo.isOperatingSystemAtLeast(_:) (Apple-documented) rather than string math. When LSMinimumSystemVersion is absent or unparseable, skip the check and allow the install (lenient — avoid blocking on missing metadata).
  • Localization. New user-facing strings go in SwiftUtilities/Utilities/en.lproj/Localizable.strings, and the resolver test SwiftUtilitiesTests/Utilities/Localization.swift should get a matching assertion.
  • Keep the Xcode project and package in sync. Both SwiftUtilities.xcodeproj and Package.swift exist. New source files must be added to the Xcode project (SwiftPM globs the target directory automatically). Build and run tests through xcodebuild per the project convention.
  • Platform guards. The skip UI is AppKit-only and lives under existing #if canImport(AppKit) / #if !SWIFT_PACKAGE guards. The deployment-target check lives in the shared UpdateInstallation (compiled into the service), so it must not depend on AppKit.

Skip suppression truth table (available update vs. stored skipped version, using isVersion(update, newerThan: skipped)):

Stored skipped versionAvailable updateOutcome (background)
noneanyShow
1.1.01.1.0 (equal)Suppress
1.1.01.0.5 (older)Suppress
1.1.01.2.0 (newer)Show (supersedes skip)

A manual "Check for Updates…" ignores the stored skip entirely and always shows an available update.

Deployment-target comparison. LSMinimumSystemVersion is a dotted string (e.g. "13.0"); parse its numeric components into an OperatingSystemVersion (majorVersion/minorVersion/patchVersion) and call ProcessInfo.processInfo.isOperatingSystemAtLeast(_:). A false result means the host is too old and the install is refused before AppReplacer runs.

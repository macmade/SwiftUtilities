In-App Updates for GitHubUpdater

Generated: 2026-07-10

Source: Docs/agent-plans/active/2026-07-10-updater/updater.html

Overview

Today GitHubUpdater only points the user at a release download: the “update available” window opens the asset’s browser_download_url or the release page in the browser via NSWorkspace. This plan adds a full in-app update path: download the new application, verify its code signature, replace the running application on disk, and relaunch it — while keeping the existing link-based behavior as the default, opt-out-free option for consumers that do not enable in-app updates.

The work is delivered as a set of small, independently testable building blocks (downloader, archive extractor, signature validator, on-disk replacer, relaunch step). Revised after M1–M5: there is a single in-app path, not separate in-process and XPC installers. Because a sandboxed app cannot replace itself, the privileged work always runs in a non-sandboxed executable, so in-app update routes through one updater XPC service bundled inside SwiftUtilities.framework (the way Sparkle 2 ships its services), which performs extract → validate → replace → relaunch off the sandbox — no consumer-built relaunch helper. Since a SwiftPM library cannot carry a signed embedded service, in-app update requires the Xcode framework + project and is compiled out of the SwiftPM build; the SwiftPM package keeps the download-link update, which remains available in every distribution. The UI reuses the existing update window, extended with a view model that drives download and install progress.

Questions & answers

How far should this plan take sandboxed-app support, given a SwiftPM library can’t embed/sign an XPC service into a consuming app?

Full XPC path now, as a single in-app mechanism for all apps (revised after M1–M5). The updater XPC service is bundled inside SwiftUtilities.framework (Sparkle 2-style), so the consumer just embeds the framework — there is no service target for them to build. Because SwiftPM cannot carry the bundled service, in-app update requires the Xcode framework; the SwiftPM package provides the download-link update only.

Before replacing the running app with a download, should the updater verify the downloaded app’s code signature?

Yes. Validate that the downloaded .app is properly code-signed and that its Team ID / signing identity matches the running app before replacing it. Without this an in-app updater is a code-execution vector.

Which release-asset archive formats should in-app update support?

Zip (.zip) and disk image (.dmg). Installer packages (.pkg) are out of scope.

How should download/install progress be presented?

Extend the existing window. Reuse UpdateAvailableView: after the user chooses to update, show progress and status in the same window, then relaunch.

Notes

Guiding principles

  • Opt-in, no regression. The current link-based flow stays the default. In-app update is enabled explicitly by the consumer; existing callers of checkForUpdates() / checkForUpdatesInBackground() keep today’s behavior unchanged.
  • Keep the pure core pure. GitHubUpdater and UpdateCheckResult in the platform-agnostic layer stay UI- and IO-free. All download/replace/relaunch code lives behind #if canImport( AppKit ), mirroring the existing split.
  • Dependency-injection seams, like the existing tests. Each building block is a protocol with an injectable implementation, so routing and pure logic are unit-testable without the network, the filesystem, or a live XPC connection — matching the existing Fetcher seam.
  • Security first. No on-disk replacement ever happens without a successful code-signature validation against a requirement derived from the running app.
  • Follow the repo conventions. public by default in the library, one type per file organized into subfolders, let over var, no force-unwrap, early returns, localized strings via Localization.string(...), SwiftUI views with previews, and the MIT license header on every new file.

Sandbox reality & distribution scope (revised)

  • Single path via a bundled service. A sandboxed app cannot replace itself, so the privileged work runs in a non-sandboxed executable — the updater XPC service. Rather than two installers, in-app update always routes through this one service, which is bundled and signed inside SwiftUtilities.framework (verified against Apple’s XPC docs and Sparkle 2’s approach in M8). The service is non-sandboxed / differently entitled so it can write to /Applications and relaunch.
  • In-app update requires the Xcode framework; the SwiftPM package is link-only and carries no extra. A SwiftPM .library can only carry a resource bundle, not a signed nested executable/XPC service. So all in-app code — the M2–M6 building blocks (archive detection, signature validation, download, extraction, replacement, the install composition) and the orchestration — is compiled out of the SwiftPM build via #if !SWIFT_PACKAGE; the building blocks are not kept as general-purpose utilities. The SwiftPM package contains only what the download-link update requires, which stays available in every distribution. (Gating is deferred to the entry-point milestone; until then the in-app code compiles in both.)
  • Mac App Store apps are out of scope for in-app updates: they update through the App Store, and MAS-embedded services must be sandboxed. Documented, not worked around.
  • Replacing an app in a protected location (e.g. /Applications) may require additional authorization. The replacement runs in the service; obtaining rights for protected directories is a risk/consideration to confirm during M8, not fully solved here.

Key existing files

  • SwiftUtilities/GitHub/GitHubUpdater.swift — platform-agnostic check logic (unchanged by this plan).
  • SwiftUtilities/GitHub/GitHubUpdater+AppKit.swift — AppKit presentation and option-to-alert routing; the in-app entry points hook in here.
  • SwiftUtilities/GitHub/UpdateCheckResult.swift — value-level result (carries downloadURL).
  • SwiftUtilities/GitHub/UpdateWindowController.swift & UpdateAvailableView.swift — the update window and its SwiftUI content, extended for progress.
  • SwiftUtilities/Windows/HostingWindowController.swift — base window controller reused by the update window.
  • SwiftUtilities/Utilities/Localization.swift and Utilities/en.lproj — localized strings; every new user-facing string is added here.
  • Tests live under SwiftUtilitiesTests/GitHub/; new tests join them.

The repository has both a Swift package (Package.swift) and SwiftUtilities.xcodeproj. Every milestone that adds or moves a source file must keep both in sync, and tests are run through xcodebuild against the Xcode project, per the project conventions.

Implementation notes

Implemented with TDD. APIs were verified against the actual Security framework SDK headers (CSCommon.h, SecCode.h, SecStaticCode.h, SecRequirement.h) rather than from memory, and the requirement-string format was confirmed against a real Developer-ID app (VS Code's designated requirement). The full Security path was additionally verified at runtime with throwaway scratchpad scripts: reading a real app's identity, a positive validation, rejection of a bogus Team ID (errSecCSReqFailed), and the SecCodeCopySelf → SecCodeCopyStaticCode → SecCodeCopySigningInformation sequence.

  • Types (all in SwiftUtilities/GitHub/): CodeSigningIdentity (pure value type, platform-agnostic) holding identifier + teamIdentifier with a pure requirement builder; CodeSignatureError (pure enum, LocalizedError); CodeSignatureInspecting (the injectable protocol seam, mirroring the Fetcher seam); SecurityCodeSignatureInspector (the real implementation, gated behind #if canImport( Security )); and CodeSignatureValidator (composes the two).
  • Requirement (the gate): identifier "<id>" and anchor apple generic and certificate leaf[subject.OU] = "<team>" — pins the signing identifier, an Apple-issued certificate chain (essential: without the anchor the Team-ID clause is spoofable), and the Team ID. Values are quoted and escaped (backslash then quote) so the requirement is injection-safe. Deliberately does not pin the Developer-ID-specific policy OIDs, so the same app+team signed through a different Apple channel still validates.
  • Root of trust: the expected requirement is derived from the running process's own validated identity via SecCodeCopySelf (revised after review; see Automated review), not from an on-disk bundle read by path. The running code is validated with SecCodeCheckValidity before its signing information is trusted, so a tampered self is refused.
  • Candidate verification: SecStaticCodeCreateWithPath + SecRequirementCreateWithString + SecStaticCodeCheckValidityWithErrors, using strict flags (kSecCSCheckAllArchitectures | kSecCSCheckNestedCode | kSecCSStrictValidate). Expiration/revocation are intentionally not enforced (matching Gatekeeper's default so a timestamped older build is not falsely rejected); this is documented in a code comment. The validator refuses on any failure and never accepts a candidate unless verification succeeds.
  • Later-milestone hooks (from review): validation targets a path, so there is a TOCTOU window between validation and the eventual on-disk replacement — the installer milestones (M4/M6/M8) must re-validate the exact bundle they move into place (and consider realpath-normalizing the path), not rely on this pre-download/pre-move check alone.

Implementation notes

Implemented with TDD. The ditto/hdiutil invocations and flags were verified against their man pages before implementing (ditto -x -k; hdiutil attach … -readonly -nobrowse -mountpoint, detach [-force], and create -srcfolder … for fixtures). End-to-end flow (in-app mode — Xcode framework only; revised)

GitHubUpdater.performUpdateCheck() → .updateAvailable( ..., downloadURL ) present( result, behavior: .inApp ) → asset is .zip / .dmg AND bundled service available ? — no → fall back to link path (SwiftPM build always lands here) — yes ↓ UpdateWindow ──▶ view model Download → UpdateDownloader (app process, progress) → XPCUpdateInstaller (client) ──▶ bundled updater XPC service (inside SwiftUtilities.framework) service: extract archive (M5) → validate signature (M3, gate — on the extracted bundle) → replace app (M6 AppReplacer, atomic same-volume) → relaunch (surviving process: wait for app PID, open new bundle)
